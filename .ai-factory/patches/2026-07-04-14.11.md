# Tooltip arrow rendering conflict during animation

**Date:** 2026-07-04 14:11
**Files:** src/components/ui/tooltip.tsx
**Severity:** medium

## Problem

Tooltip arrow rendered unstably during animation — it jittered for approximately 500ms when tooltip appeared, then settled to stable position. Visual jitter made the tooltip feel janky and broken.

## Root Cause

The `TooltipPrimitive.Arrow` component had manually applied transforms:
- `translate-y[calc(-50%_-_2px)]` — CSS transform adjusting vertical position
- `rotate-45` — CSS transform rotating 45 degrees

During parent's `animate-in` (slide-in-from-top, zoom-in, fade-in) animation, these arrow transforms conflicted with the parent animation, causing the arrow to appear unstable. Radix UI's tooltip library automatically positions arrows using CSS variables and internal positioning logic — manual transforms interfered with this.

The developer added manual transforms thinking they were needed for arrow positioning, but Radix UI handles this automatically.

## Solution

Removed conflicting manual transforms:
```tsx
// Before
<TooltipPrimitive.Arrow className="z-50 size-2.5 translate-y-[calc(-50%_-_2px)] rotate-45 rounded-none bg-foreground fill-foreground" />

// After
<TooltipPrimitive.Arrow className="z-50 size-2.5 rounded-none bg-foreground fill-foreground will-change-auto motion-safe:will-change-transform" />
```

Added `will-change-auto` and `motion-safe:will-change-transform` for rendering optimization during animation.

## Prevention

- **For Radix UI components:** Never manually apply transforms to positioned primitives (Arrow, Popover.Content, etc.). Radix library handles positioning — override only styling (color, size, border-radius).
- **For compound animations:** Test tooltip/popover/dialog animations on hover to catch jitter during appearance.
- **Code review check:** When reviewing Tooltip/Popover/Dialog wrapper components, verify that positioning primitives (Arrow, Content) don't have conflicting `translate-*`, `rotate`, or `scale` classes that would override library-managed positioning.
